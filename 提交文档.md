## 项目概述

简单描述项目思路、技术实现、部署环境、项目亮点 / 创新点等信息。

### 1. 项目思路

短剧用户的情绪峰值通常出现在冲突、反转、打脸、悬念、甜蜜撒糖等剧情高光点。传统播放器只负责播放视频，无法在这些关键时刻激发用户即时互动。本项目的核心思路是：先用大模型理解短剧内容并生成剧情高光，再通过审核后台完成人工校验和发布，最后在播放器端以进度条高光点、摘要卡片、互动按钮、全屏动效和互动统计的方式触发用户互动。

项目形成了一个完整闭环：真实 MP4 短剧进入系统后，AI 负责生成候选高光，审核后台负责修正和发布，前台播放器负责端上可视化互动，用户行为再回流到后端形成统计数据，并进一步支持 AIGC 剧情分支创作。

### 2. 当前已实现内容

当前项目已经完成可演示 MVP，并额外补充了真实数据库后端版本。前台支持真实 MP4 剧集播放、剧集搜索、选集、高光进度条、全屏响应式、互动按钮和互动统计。审核后台支持按剧集管理高光，提供高光查看、编辑、发布、删除、批量操作、视频预览、解析进度展示、AIGC 分支管理和视频生成任务池。

后端目前有两套实现：Node.js 版本用于快速演示，负责静态资源、媒体流、模型调用、高光导入和 JSON 数据保存；Django 版本用于正式 Web 项目结构，提供 ORM 模型、数据库 API、JSON 数据导入命令和运行时数据库写入能力。两套服务互不破坏，Node 保证演示稳定，Django 保证真实数据库持久化链路。

### 3. 产品亮点与创新点

项目不是简单做一个视频播放器，而是把“剧情理解”和“即时互动”结合起来。高光不是人工配置的固定时间点，而是由大模型对短剧内容进行理解后生成候选结果，再经过审核后台人工确认，降低模型误判风险。

AIGC 分支生成也不是对整集剧情泛化续写，而是基于某个被选中的具体高光点生成。这样分支剧情更贴合用户情绪峰值，也更适合后续生成分支视频或互动剧情选项。

### 4. 技术关键点

前端采用视频层、动效层、互动层、控制层分离设计。高光点嵌入进度条内部，动效层悬浮在视频上方，不修改原视频内容。全屏模式下播放器、进度条和按钮做了响应式适配。

Node 后端负责快速跑通演示闭环，Django 后端负责真实数据库版本。Django 通过模型层保存短剧、剧集、高光、互动、AIGC 分支、视频生成任务和模型运行记录，前端行为可以通过 API 真实写入数据库。

### 5. 部署环境

本地演示可以运行 Node 服务：

```powershell
node server.js
```

真实数据库版本可以运行 Django 服务：

```powershell
cd backend_django
python manage.py runserver 127.0.0.1:8000
```

正式部署建议采用 Nginx + Django + MySQL + 对象存储 / CDN。MP4 文件不直接存入 MySQL，数据库只保存 `video_url`、`local_path`、文件大小、时长、封面和状态等元信息。

### 1. 模块分析和拆解

前台播放器模块：负责剧集搜索、选集、视频播放、播放 / 暂停、音量、全屏、进度条高光点、剧情摘要卡片、互动按钮、互动统计和高光动效展示。

审核后台模块：负责按剧集查看高光，支持高光预览、编辑、发布、删除、批量操作、视频预览、解析进度展示、AIGC 分支管理和视频生成任务池管理。

Node 演示后端模块：负责现有前端页面托管、MP4 Range 媒体流、高光 CRUD、互动写入、模型解析、AIGC 分支生成、视频任务保存和本地 JSON 数据读写。

Django 数据库后端模块：负责真实数据库版本的模型设计和 API 实现。Django 中已实现短剧、剧集、高光、互动、AIGC 分支、视频生成任务、模型运行记录和配置表，前端行为可以通过 API 写入 SQLite 或 MySQL。

数据迁移模块：提供 `docs/mysql_schema.sql`、`scripts/export_db_json_to_mysql_sql.js`、`data/mysql_seed.sql` 和 Django 的 `import_project_data` 命令，用于把已有 JSON 和 MP4 剧集索引导入数据库。

AI 能力模块：Node 版本通过模型配置和 Python 脚本调用 Doubao-Seed-2.0-lite，生成结构化高光。Django 版本提供数据库写入和兜底解析接口，正式部署时可继续把 Doubao 调用迁移到 Django service 中。

### 3. 核心模块技术选型和理由

前端选择原生 HTML / CSS / JavaScript，原因是项目周期短，原生技术可以减少框架依赖，保证播放器控制、全屏、进度条和动效逻辑可控。

Node.js 后端用于快速演示，原因是它能以较少依赖实现静态资源托管、媒体流、JSON 数据读写和模型调用，非常适合 MVP 快速交付。

Django 后端用于真实数据库落地，原因是 Django ORM、模型层和管理命令更适合构建正式 Web 项目。当前 Django 后端已经可以把前端行为通过 API 写入数据库，并提供 JSON 数据导入能力。

数据库选择 MySQL，原因是短剧、剧集、高光、互动、任务、配置之间存在明确关系，适合用关系型数据库管理。视频文件本身不存入数据库，只保存访问地址和元信息。

### 4. 当前数据保存逻辑

Node 演示版本中，高光、互动、AIGC 分支等数据保存到 `data/db.json`，旧数据兼容 `data/store.json`，视频生成任务保存到任务池文件，生成视频保存到 `data/generated`。

Django 数据库版本中，前端请求直接进入 Django API，再由 ORM 写入数据库。用户点击互动按钮会写入 `interactions` 表；后台新增或编辑高光会写入 `highlights` 表；生成 AIGC 分支会写入 `video_branches` 表；提交视频生成任务会写入 `video_generation_tasks` 表。

已有 JSON 数据可以通过以下命令导入 Django 数据库：

```powershell
cd backend_django
python manage.py migrate --run-syncdb
python manage.py import_project_data
```

如果要使用 MySQL，可以先设置环境变量：

```powershell
$env:MYSQL_DATABASE="juran_moment"
$env:MYSQL_USER="root"
$env:MYSQL_PASSWORD="你的密码"
$env:MYSQL_HOST="127.0.0.1"
$env:MYSQL_PORT="3306"
```

然后执行同样的迁移和导入命令。

### 5. 大模型 / AI 能力使用说明

项目围绕 Doubao-Seed-2.0-lite 的多模态理解能力设计，重点使用其音画联合理解、视频事件时间戳定位和跨时间段剧情追踪能力。模型输入是真实短剧视频和 Prompt，输出为结构化高光 JSON。

高光字段包括：剧集 key、开始时间、结束时间、高光类型、情绪、强度、置信度、摘要、互动建议、模型来源、模型理由和状态。后端会将这些字段标准化后写入数据层，审核后台再进行人工修正和发布。

AIGC 分支生成基于“选中高光”触发。系统不会对整集无差别生成分支，而是把某个高光的摘要、类型、情绪和互动建议作为上下文，生成分支剧情、用户选择文案和视频生成 Prompt。分支内容只在审核后台管理，前台不直接展示 AIGC 文本，保证观看体验干净。

### 7. 工程难点与解决方案

难点一：AI 输出不稳定。解决方案是通过 Prompt 约束模型输出 JSON，再在后端进行字段归一化，最后通过审核后台人工确认后发布。

难点二：播放器全屏、进度条和动效容易互相干扰。解决方案是将视频层、动效层、互动层、控制层拆开，高光点嵌入进度条内部，全屏下重新适配布局。

难点三：本地 JSON 演示和真实数据库落地之间存在断层。解决方案是保留 Node 演示后端，同时新增 Django 数据库后端和导入命令，使已有 JSON 数据可迁移，运行时行为可写库。

难点四：分支视频可能被误识别为原始剧集。解决方案是将 `data/generated` 明确标记为非剧集目录，Node、Django 和 SQL 导出链路都不会把生成视频当成普通短剧剧集。

## 项目总结和自评

简要的对照课题进行总结和自评，思考收获以及不足之处。

### 1. 对照课题完成情况

项目已经覆盖题目要求中的短剧内容理解、互动形式设计和可视化、后台服务模块与接口、状态管理和可扩展性设计。AI 高光识别为即时互动提供基础数据；前台播放器负责高光可视化和互动触发；审核后台负责人工校验、发布和 AIGC 分支管理；Django 后端补齐了真实数据库保存链路。

### 2. 收获

本项目验证了一个重要思路：短剧互动不应该只停留在弹幕或点赞，而应该围绕剧情情绪峰值触发。高光点比整集摘要更适合作为互动入口，也更适合作为 AIGC 分支创作入口。

工程上，项目从一开始的本地 JSON 演示逐步扩展到 Django 数据库后端，形成了从 MVP 到真实 Web 项目的过渡方案。这让项目既能快速演示，也能说明后续部署和扩展路径。

### 3. 不足

当前 Django 版本已经具备数据库写入和导入能力，但 Doubao 模型调用主要仍在 Node 演示链路中，后续可以进一步迁移到 Django service 层。

视频生成任务池目前以任务展示和接口预留为主，正式上线仍需要接入稳定的视频生成模型 API，并处理异步回调、失败重试、成本控制和内容审核。

Android 端目前是 WebView 壳，已经可以快速展示 Web 项目，但距离完整原生端体验仍需继续优化手势、缓存、沉浸式全屏和弱网策略。

### 4. 后续优化方向

后续可以继续完善 MySQL repository、Redis 缓存、对象存储 / CDN、Lottie / Canvas / WebGL 动效包、移动端适配、视频生成模型接入和高光推荐算法。

在业务上，可以进一步基于互动数据做热门高光排行、用户偏好分析、AIGC 分支优先级排序和短剧内容运营看板，让系统从“可演示 MVP”升级为更完整的内容互动平台。
