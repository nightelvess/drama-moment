## 项目概述

简单描述项目思路、技术实现、部署环境、项目亮点 / 创新点等信息。

### 1. 项目思路

本项目围绕短剧播放中的“剧情高光时刻”设计即时互动体验。系统先使用大模型理解真实短剧 MP4 内容，识别冲突、反转、爽点、甜蜜、名场面等高光片段，再通过审核后台进行人工修正和发布，最终在前台播放器中以进度条高光点、剧情摘要卡片、互动按钮、全屏动效和互动统计的形式触发用户互动。

### 2. 核心功能介绍

已实现真实剧集播放、剧集搜索与选集、AI 高光解析、后台高光审核、视频预览、批量操作、冲突检测、前台高光动效、互动数据统计、按选中高光生成 AIGC 分支、视频生成任务池展示，以及 MySQL 迁移脚本。

AIGC 分支内容只在审核后台展示和管理，前台保持纯观看与互动体验，避免剧情分支文本干扰短剧播放。

### 3. 产品亮点与创新点

项目把“短剧内容理解”和“端上即时互动”连接成闭环，不是单纯播放视频。用户在剧情情绪峰值处完成互动表达，互动数据再回流到后端用于热度统计。

AIGC 分支不是整集泛化续写，而是基于某个具体高光点生成，更贴合用户对关键剧情的二次想象。

### 4. 技术关键点

前端采用视频层、动效层、互动层、控制层分离的播放器结构，高光点嵌入进度条内部，全屏下进行响应式适配。

后端使用 Node.js 提供静态资源、媒体流、数据 API、模型调用入口和任务池接口。模型调用通过 Python 脚本读取配置并请求 Doubao-Seed-2.0-lite，输出结构化高光数据。

### 5. 部署环境

当前开发环境为 Windows 本地 Node.js 服务，默认访问 `http://127.0.0.1:3000/`。

部署到服务器时建议使用 Node.js + Nginx + MySQL。MP4 文件建议放在服务器磁盘、对象存储或 CDN 中，数据库只保存 URL、路径、封面、时长、大小等元信息。

### 1. 模块分析和拆解

前台播放器模块：负责剧集播放、搜索选集、播放 / 暂停、全屏、进度条高光点、剧情摘要卡片、互动按钮、动效层和互动统计展示。

审核后台模块：负责按剧集管理 AI 生成的高光，支持查看、编辑、发布、删除、批量操作、视频预览、时间冲突检测和 AIGC 分支管理。

后端服务模块：负责静态资源服务、MP4 媒体流服务、剧集 API、高光 API、互动 API、模型解析 API、AIGC 分支 API 和视频生成任务 API。

AI 解析模块：负责调用 Doubao-Seed-2.0-lite 对短剧视频进行高光理解，并输出结构化结果。

数据存储模块：本地使用 JSON 快速迭代，运行时通过后端统一读写 `data/db.json`，并提供 MySQL 建表脚本和种子数据导出脚本，方便部署阶段迁移。

### 2. 整体流程图

```text`n真实短剧 MP4 -> 剧集加载与媒体流服务 -> AI 高光理解 -> 结构化高光数据 -> 审核后台 -> 前台播放器 -> 用户互动 -> 互动统计回流 -> AIGC 分支 -> 视频生成任务池 -> MySQL 数据层`n```

文字版流程：真实短剧 MP4 先进入剧集索引和媒体流服务，后台调用 Doubao-Seed-2.0-lite 解析剧情高光，生成带时间戳的结构化高光数据。高光数据先写入本地数据层 `data/db.json`，审核人员在后台预览、编辑、冲突检测和发布高光后，前台播放器在对应时间点展示进度条高光点、摘要卡片、互动按钮和动效。用户互动数据继续回写到 `data/db.json` 并形成统计，同时审核人员可以基于某个选中高光生成 AIGC 剧情分支，并把 `video_prompt` 提交到视频生成任务池。部署到服务器时，通过 MySQL schema 和 seed SQL 将本地数据迁移到正式数据库。

### 3. 当前数据保存逻辑与数据库入库方案

当前 MVP 运行时采用“后端统一写 JSON，本地可导出 MySQL”的方式保存数据。这样做的原因是开发和演示阶段迭代速度快，不需要依赖本地 MySQL 服务；同时项目已经准备好 MySQL 建表和数据导出脚本，便于部署阶段迁移。

当前运行时保存逻辑如下：

1. 剧集数据：后端扫描 `data` 目录下的 MP4 文件夹，生成剧集索引；MP4 文件本身不写入数据库，只保存剧集 key、路径、文件名、播放地址、时长、大小等元信息。
2. AI 高光数据：后台点击解析高光后，后端调用模型脚本得到结构化高光，再通过 `addHighlights()` 写入 `data/db.json` 的 `highlights` 数组；编辑、发布、删除高光时也会更新同一个数据源。
3. 用户互动数据：前台点击互动按钮后，请求 `/api/interactions`，后端生成互动记录并写入 `data/db.json` 的 `interactions` 数组，同时更新统计字段，用于前台互动统计和 activity feed。
4. AIGC 分支数据：后台对选中高光生成分支后，后端将分支包写入 `data/db.json` 的 `videoBranches` 数组，每个分支保留触发高光、剧情节点、用户选项、视频生成 Prompt 和模型原始输出。
5. 视频生成任务：提交 `video_prompt` 后，任务信息保存为任务池数据，包含模型、状态、Prompt、生成视频地址或本地文件路径；生成的 MP4 文件放在 `data/generated`，数据库只记录文件路径和访问 URL。
6. 模型配置和动效配置：模型参数保存到 `data/model-settings.json`，高光动效配置保存到 `data/effects-config.json`，避免把 API Key 和动效策略写死在代码中。

MySQL 入库方案如下：

1. 先执行 `docs/mysql_schema.sql` 创建正式数据库表，表包括 `dramas`、`episodes`、`highlights`、`interactions`、`model_runs`、`continuations`、`video_branches`、`video_generation_tasks`、`app_settings` 等。
2. 再执行 `scripts/export_db_json_to_mysql_sql.js`，脚本会读取 `data/db.json`、`data/store.json` 和本地 MP4 文件目录，生成 `data/mysql_seed.sql`。
3. 最后将 `data/mysql_seed.sql` 导入 MySQL，把当前剧集元信息、高光、互动、AIGC 分支和视频任务数据写入数据库。

导入命令示例：

```powershell
node scripts\export_db_json_to_mysql_sql.js
mysql --default-character-set=utf8mb4 -u root -p < docs\mysql_schema.sql
mysql --default-character-set=utf8mb4 -u root -p < data\mysql_seed.sql
```

正式部署时建议将当前 JSON 读写函数抽象为 repository 层，例如 `HighlightRepository`、`InteractionRepository`、`VideoBranchRepository`。本地演示环境继续使用 JSON repository，服务器环境切换为 MySQL repository。这样可以保证接口不变，前端和模型调用逻辑不需要重写，只替换数据持久化实现。

### 4. 核心模块技术选型和理由

前端选择原生 HTML / CSS / JavaScript，减少依赖并快速稳定实现播放器和审核后台。

后端选择 Node.js 原生 HTTP 服务，便于实现 REST API、静态资源托管和 MP4 Range 请求。

AI 模型选择 Doubao-Seed-2.0-lite，利用其视频、图像、音频、文本统一理解能力完成音画联合理解、时间戳定位和剧情高光抽取。

数据层采用 JSON 本地存储 + MySQL 迁移脚本，兼顾开发效率和部署可迁移性。

### 5. 大模型 / AI 能力使用说明

模型配置位于 `data/model-settings.json`，包括 API Key、Endpoint、模型名称等参数。

高光解析通过 `scripts/analyze_episode.py` 发起模型请求，Prompt 要求模型返回严格 JSON，核心字段包括高光类型、起止时间、情绪、强度、置信度、摘要、互动建议和模型理由。

AIGC 分支生成不直接面向前台用户开放，而是在审核后台由人工选择某个高光后触发。系统会把该高光的剧情摘要、类型、情绪和互动建议作为上下文，生成多个剧情分支和对应的视频生成 Prompt。

### 6. 工程难点与解决方案

1. 模型输出不稳定：使用 Prompt 约束输出结构，并在后端做字段归一化；上线前通过审核后台人工编辑和发布。
2. 播放器全屏和进度条容易混乱：将视频层、动效层、互动层、控制层拆分，高光点嵌入进度条内部，全屏下重新适配布局，播放 / 暂停统一为一个按钮。
3. AIGC 视频生成链路成本高：先生成分支剧情和视频 Prompt，放在后台任务池管理，不让前台用户直接触发高成本视频生成。

## 项目总结和自评

简要的对照课题进行总结和自评，思考收获以及不足之处。

### 1. 对照课题完成情况

项目已覆盖题目要求中的短剧内容理解、互动形式设计和可视化、后台服务流程与闭环设计。

短剧内容理解通过 Doubao-Seed-2.0-lite 完成高光识别；端上渲染通过播放器进度条高光点、全屏动效、互动按钮和统计展示完成；后台闭环通过审核、发布、互动回流和 AIGC 分支任务池完成。

### 2. 收获

本项目把大模型能力从“离线分析”落到“内容平台可用流程”中，形成了 AI 解析、人工审核、端上触发和数据回流的完整链路。

同时也验证了短剧场景下高光点比整集摘要更适合作为互动和 AIGC 创作入口。

### 3. 不足

当前运行时仍以 JSON 存储为主，MySQL 已提供迁移脚本但尚未完全替换运行时数据层。

视频生成任务池仍是演示版，需要接入稳定的视频生成模型 API。

Android 端目前是 WebView 壳，距离完整原生端体验仍需继续优化。

### 4. 后续优化方向

接入正式 MySQL repository 层；扩充 Lottie / Canvas / WebGL 动效包；完善移动端手势和沉浸式全屏；增加线上部署、对象存储和 CDN；基于互动数据优化高光排序和 AIGC 分支推荐。
